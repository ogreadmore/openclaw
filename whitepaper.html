Executive Summary

OpenClaw is an open-source AI agent platform that gives autonomous bots full system access: shell commands, file systems, browser control, and messaging APIs. That power makes it extraordinarily useful. It also makes it extraordinarily dangerous when deployed with default settings.

Out of the box, OpenClaw ships with authentication disabled, no firewall, and the gateway exposed to the public internet. Security researchers at Kaspersky, Palo Alto Networks, and Cisco Talos have all flagged default OpenClaw installations as high-risk targets for data exfiltration, prompt injection, and unauthorized remote access.

ManagedOpenClaw exists to close that gap. Every deployment we ship is hardened from the operating system up, isolated on dedicated infrastructure, and configured with defense-in-depth security controls. This white paper explains exactly what we do and why.

The Problem with Default OpenClaw

OpenClaw's maintainers are transparent about the risks. The onboarding wizard itself displays a security warning that reads, in part: "OpenClaw is a hobby project and still in beta. Expect sharp edges... A bad prompt can trick it into doing unsafe things."

The default installation has several critical security gaps:

  • No authentication. The gateway dashboard and API are accessible to anyone who can reach the server. There is no login, no token, no access control of any kind.
  • Gateway bound to all interfaces. The web gateway listens on 0.0.0.0, meaning it accepts connections from every network interface, including the public internet.
  • No firewall. There is no firewall configured. Every port on the server is reachable from the internet unless the hosting provider has a separate network-level firewall.
  • No encryption in transit. The gateway communicates over plain HTTP. Credentials, tokens, and conversation data are transmitted in cleartext.
  • Full tool access by default. The agent has unrestricted access to shell commands, file read/write, and network requests. A prompt injection attack can leverage all of these capabilities.
  • SSH with password login. Most cloud providers ship Ubuntu images with password-based SSH enabled and root login permitted.

For a developer running OpenClaw locally on their laptop behind a NAT, these defaults are tolerable. For a business deploying it on a public server to serve customers, they are not.

Threat Landscape

AI agents with tool access represent a new category of attack surface. The risks are not theoretical. The security community has documented several classes of attack against systems like OpenClaw:

Prompt Injection

An attacker crafts input (via a messaging channel, uploaded document, or web request) that causes the agent to execute unintended actions. With tool access enabled, this can mean reading sensitive files, executing shell commands, or exfiltrating data to an external server.

Credential Theft

If the gateway is accessible without authentication, an attacker can access the dashboard, read conversation history, extract API keys visible in configuration, and issue commands to the agent directly.

Lateral Movement

An agent with shell access on an unpatched server can be used as a pivot point. If the server has access to internal networks, databases, or other services, the agent becomes an entry point for deeper compromise.

Data Exfiltration

Conversation history, uploaded documents, customer data processed by the agent, and configuration files (including API keys) are all accessible to anyone who can reach an unprotected gateway.

The core issue: OpenClaw gives your agent the same level of access as a human sitting at the server's terminal. Without security controls, anyone on the internet has that same access.

Our Hardening Approach

ManagedOpenClaw applies defense-in-depth security: multiple independent layers of protection so that no single failure compromises the system. Every deployment includes the following controls, configured automatically during provisioning.

1. Operating System Hardening

Every server begins with a minimal Ubuntu LTS image. We immediately apply the following:

  • SSH key-only authentication. Password login is disabled. Root login is disabled. Only the designated service user can authenticate, and only with a pre-authorized SSH key.
  • Firewall with explicit allowlist. UFW (Uncomplicated Firewall) is configured with a default-deny policy. Only ports 22 (SSH), 80 (HTTP for TLS certificate issuance), and 443 (HTTPS) are open. All other ports are blocked.
  • Brute-force protection. fail2ban monitors SSH logs and automatically bans IP addresses after repeated failed authentication attempts.
  • Automatic security updates. Unattended-upgrades is enabled for the security pocket. Critical OS patches are applied automatically without manual intervention.

2. Network Isolation

The OpenClaw gateway runs inside a Docker container. Docker has a well-known behavior where published ports bypass the host firewall (UFW/iptables) by manipulating iptables rules directly. This means that even with UFW configured, a Docker container publishing a port can be reachable from the internet.

We solve this at the Docker daemon level. The daemon is configured to bind all published ports to 127.0.0.1 (localhost) by default. This means no Docker container on the server can expose a port to the public internet, regardless of what the compose file specifies. The only path to the gateway from the outside world is through the Caddy reverse proxy on port 443.

3. Encrypted HTTPS via Caddy

Caddy is a production-grade reverse proxy that automatically provisions and renews TLS certificates from Let's Encrypt. All traffic between the client's browser and the gateway is encrypted. There is no plain HTTP access to the gateway.

Caddy serves as the single ingress point. It terminates TLS, enforces HTTPS, and forwards traffic to the gateway on the local loopback interface. The gateway itself is never directly reachable from the internet.

4. Gateway Authentication

Every deployment is configured with a cryptographically random 256-bit gateway token generated during provisioning. The dashboard and API require this token for access. Without it, the gateway rejects all connections.

Additionally, OpenClaw's device pairing system is enforced. Each browser that accesses the dashboard must be explicitly approved from the server's command line. This provides a second factor of access control beyond the token itself.

5. Docker Container Isolation

The OpenClaw application runs inside a Docker container with its own filesystem, process namespace, and network stack. The container runs as a non-root user with a unique UID that differs from the host system. Configuration files are shared via Docker volume mounts with carefully managed permissions: world-readable for files the container needs to access, with sensitive credentials (OAuth tokens) locked to owner-only access (600).

6. CORS and Origin Restrictions

The gateway's Control UI is configured with an explicit allowedOrigins list that only permits requests from the deployment's HTTPS subdomain. Cross-origin requests from other domains are rejected, preventing clickjacking and cross-site request forgery against the dashboard.

Deployment Architecture

Every ManagedOpenClaw deployment follows this architecture:

  • Dedicated VPS — one server per client, provisioned on Hetzner Cloud (or AWS/DigitalOcean on request). No shared infrastructure, no multi-tenancy.
  • Ubuntu LTS — minimal image with automatic security updates.
  • Docker + Docker Compose — application runs in isolated containers. Daemon configured for localhost-only port binding.
  • Caddy reverse proxy — automatic HTTPS, TLS termination, single ingress point on ports 80/443.
  • UFW firewall — default-deny inbound. Ports 22, 80, 443 only.
  • fail2ban — SSH brute-force protection.
  • Health monitoring — automated health checks every 5 minutes with automatic restart on failure.
  • Encrypted backups — nightly restic backups to off-server storage with 30-day retention. AES-256 encryption at rest. Included with managed care plans.

Ongoing Security Operations

Initial hardening is necessary but not sufficient. Security is an ongoing practice, not a one-time configuration. Our managed care plans include:

  • Same-day critical patches. When a CVE with a CVSS score of 9.0 or higher is published for any component in the stack (Ubuntu, Docker, Caddy, Node.js, OpenClaw), we apply the patch the same day it becomes available.
  • Routine updates within 48 hours. Non-critical OpenClaw updates, dependency patches, and minor version bumps are applied within 48 hours of stable release.
  • Weekly health checks. We review uptime, container status, disk usage, and gateway logs weekly. Issues are flagged and resolved before they become incidents.
  • Monthly backup restore tests. Backups are only useful if they work. We test a full restore from backup monthly and verify the gateway comes up correctly.
  • API cost monitoring. We set usage alerts on your AI model API spend. If costs spike unexpectedly (a signal of abuse or misconfiguration), you're notified immediately.

The Ownership Model

ManagedOpenClaw is not a SaaS platform. We do not host your agent on our infrastructure. We do not hold your API keys. We do not have ongoing access to your server unless you explicitly grant it for managed care.

Every deployment is built on infrastructure that you own. The VPS is in your cloud account (or a dedicated account we create for you). The domain points to your server. The API keys are yours. If you stop working with us, everything keeps running exactly as it was.

This model exists because we believe that businesses running AI agents with access to sensitive data should own and control the infrastructure those agents run on. Multi-tenant platforms are fine for experimentation. For production use with client data, employee data, or proprietary workflows, isolated, owned infrastructure is the appropriate security posture.

Conclusion

OpenClaw is a powerful platform. Its default security posture is designed for local development, not production deployment. The gap between those two contexts is significant, and closing it requires expertise across operating system hardening, container security, network architecture, and TLS configuration.

ManagedOpenClaw closes that gap for every client, every deployment, from day one. If you have questions about our security practices or want to discuss your specific requirements, we welcome the conversation.
